## Что делает Doba

Приложение в **меню-баре** macOS (основное) + **интерактивный виджет на рабочем столе** (лёгкая поверхность «глянул план — отметил сделанное»). Оба таргета делят одни локальные данные через **App Group**.

Целевая ОС — **последняя macOS** (предыдущие версии не поддерживаем), так что интерактивный десктоп-виджет доступен гарантированно.

Функции:
- Список задач на сегодня с чекбоксами (done / not-done) и **проект-тегом** — много разных проектов, нужно видеть распределение внимания.
- **Четыре независимые оси у задачи** (не смешивать в один флаг):
  1. **Привязка ко времени.** Часть задач прибита к слоту («обязательные со временем»), часть плавающие на день («просто надо сделать»). Поле `scheduledTime: Date?`: задано — задача на таймлайне рядом с митами; nil — в пуле плавающего чеклиста.
  2. **План vs факт.** `estimatedHours` — сколько заложено; `actualHours` — сколько реально ушло.
  3. **Billable.** Флаг: оплачиваемая задача или важная-но-неприбыльная (overhead).
  4. **Статус.** todo / done.
- **Таймер и учёт времени.** Один активный таймер за раз (работаю над одним делом — так нет двойного учёта). Старт/стоп на задаче копит `actualHours`. Плюс ручная правка часов, если забыл нажать старт.
- Дневная свёртка по проектам и по типам: «сегодня запланировано 6ч на X, 2ч на Y + 2 мита (3ч); потрачено 4ч; billable 5ч / overhead 3ч». По сути встроенный таймшит фрилансера.
- Митинги тянутся **read-only из системного `Calendar.app` через EventKit** (там уже синкан мой Google-календарь по email). Никакого Google API / OAuth / Vercel.
- Невыполненные задачи прошлых дней **автоматически переносятся на сегодня** при запуске / смене дня, помечаются как carried-over.
- Today-view делится на: **таймлайн** (миты + задачи с `scheduledTime`) и **плавающий пул** (задачи без времени).
- Быстрый ввод: текстовое поле (системная диктовка macOS `fn fn` работает в нём из коробки) → кнопка Parse → **Claude API** превращает естественный текст в структурированные задачи (заполняет проект, часы, время, billable). Поддержать и одиночные («подготовить эстимейт, 2ч, оплачиваемо»), и диапазонные («таск X, 6ч/день с пн по пт»).

## Зафиксированные решения (не предлагай альтернатив)

- **Натив, SwiftUI.** Не web, не Tauri, не Electron.
- **Mac-only.** Никакого iPhone-таргета, никакого синка.
- **Хранение — локальный JSON-файл в контейнере App Group** через тонкий `Codable`-store. Данных мало, нужна предсказуемость и отлаживаемость на первом Swift-проекте. SwiftData — кандидат на v2, не сейчас.
- **Никакой удалённой БД / Vercel / backend.** Это однопользовательский локальный инструмент.
- **Календарь — EventKit, read-only.** События не храним, мерджим в today-view на лету.
- **UI — строго нативный по macOS Human Interface Guidelines.** Системные материалы / vibrancy, SF Symbols, стандартные контролы и `MenuBarExtra` / WidgetKit-конвенции (фиксированные размеры виджет-семейств). Не изобретай кастомные контролы и не тащи веб-паттерны (брейкпоинты, адаптивность) — их тут нет.
- **NL-парсинг — Claude API.** Для парсинга используем `claude-haiku-4-5-20251001` (дёшево и достаточно); если качество разбора диапазонов будет хромать — поднимем до `claude-sonnet-4-6`. Системный промпт обязывает модель вернуть **только JSON** по схеме задачи, без преамбулы и без ```-ограждений; ответ парсим безопасно.
- **Секреты не коммитим.** API-ключ — в Keychain или в gitignored-конфиге, никогда в репозитории.

## Архитектура

Три модуля в одном Xcode-проекте:
1. **DobaApp** — `MenuBarExtra`-приложение. Тяжёлая логика: ввод, парсинг, ведение задач, чтение календаря, перенос несделанного, свёртка.
2. **DobaWidget** — WidgetKit extension. Интерактивный виджет (macOS 14+, App Intents для тоглов чекбоксов прямо в виджете). Только читает общий store и тикает галочки; данными не управляет. После любой записи в store приложение дёргает `WidgetCenter.shared.reloadAllTimelines()`.
3. **DobaKit** — общий Swift package / framework: модели, store, бизнес-логика. Используется обоими таргетами, чтобы не дублировать.

Поток данных: ввод/парсинг/календарь → DobaKit store (JSON в App Group) → читают и меню-бар, и виджет.

## Модель данных (v1)

- `Project` — лёгкий тег: `id`, `name`, `colorHex`. Опционален у задачи.
- `Task` — атомарная единица **на конкретный день**: `id`, `title`, `projectID?`, `scheduledDate` (день), `scheduledTime: Date?` (nil = плавающая, задано = привязана к слоту), `estimatedHours: Double?`, `billable: Bool`, `status` (`todo` / `done`), `isCarriedOver: Bool`, `notes?`, `source` (`manual` / `dictated` / `parsed`), `createdAt`. `actualHours` **не хранится в задаче напрямую**, а выводится из `TimeEntry` (см. ниже).
- `TimeEntry` — запись отрезка работы: `id`, `taskID`, `start: Date`, `end: Date?`. `end == nil` означает «таймер идёт прямо сейчас». `actualHours` задачи = сумма длительностей её завершённых записей (+ живой отсчёт у текущей). Источник правды по факту-часам — именно записи, не одно поле, чтобы учёт был честным и редактируемым.
- **Один активный таймер на всё приложение.** Старт = создать `TimeEntry` с `end == nil` (если уже есть активный по другой задаче — сперва закрыть его). Стоп = проставить `end`. Состояние «какой таймер активен» выводится из наличия записи с `end == nil`, отдельного флага не надо.
- **Диапазонные задачи разворачиваем в N атомарных `Task` (по одной на день диапазона), каждая со своими `estimatedHours`.** Сознательный размен: тривиальные свёртка, чекофф, перенос и учёт часов по дням ценой дублирования заголовка. Альтернатива (`PlannedBlock` + развёртка на лету) — кандидат в DECISIONS как опция для v2.
- Митинги **не модель** — это `EKEvent` из EventKit, мерджатся в today-view только для отображения и для часов в свёртке.

Свёртки дня:
- **Запланировано** = сумма `estimatedHours` сегодняшних задач по проектам + суммарная длительность сегодняшних событий календаря.
- **Потрачено** = сумма `actualHours` (из `TimeEntry`) за сегодня.
- **Billable vs overhead** = разбивка часов (план и факт) по флагу `billable`.

Перенос: при запуске / смене суток у всех `Task` где `scheduledDate < today && status == .todo` → `scheduledDate = today`, `isCarriedOver = true`. `TimeEntry` прошлых дней не трогаем — история факта остаётся за своей датой.

## Фазы

- **Phase 0 — Scaffold (делаем сейчас).** Xcode-проект; три модуля (DobaApp, DobaWidget, DobaKit); настроенный App Group; все доки и CLAUDE.md; `.gitignore` (Xcode + секреты); компилируемый скелет с парой dummy-задач в меню-баре и в виджете. **Подписывание / включение App Group в capabilities в Xcode — выпиши мне отдельным чеклистом ручных шагов, это я делаю руками.** После Phase 0 — стоп и ревью.
- **Phase 1 — Core + меню-бар.** JSON-store в App Group, модели `Task` / `Project` со всеми осями (время / план / billable / статус), панель `MenuBarExtra` с today-view: таймлайн (привязанные ко времени) + плавающий пул, добавить / отредактировать / чекнуть. Логика переноса несделанного.
- **Phase 2 — Календарь.** EventKit read-only (запрос доступа один раз), мердж сегодняшних событий в таймлайн, свёртка **запланированных** часов (задачи по проектам + митинги).
- **Phase 3 — Таймер и учёт времени.** Модель `TimeEntry`, один активный таймер (старт/стоп на задаче), вывод `actualHours`, ручная правка часов. Свёртки **план vs факт** и **billable vs overhead**.
- **Phase 4 — NL-парсинг.** Текстовое поле → Claude API → структурированные задачи (одиночные и диапазонные), заполнение проекта / часов / времени / billable. Диктовка работает бесплатно через фокус на поле.
- **Phase 5 — Виджет.** Интерактивный WidgetKit-виджет: сегодняшний план + чекбоксы через App Intents, читает общий store, релоадится после записей приложения. (Управление таймером оставляем в меню-баре; кнопка таймера в виджете — кандидат в полировку.)
- **Phase 6 — Полировка (потом).** Глобальный хоткей для диктовки, ставки и подсчёт заработка по billable-часам, старт таймера из виджета, цвета проектов, возможная миграция на SwiftData, повторяющиеся задачи.
